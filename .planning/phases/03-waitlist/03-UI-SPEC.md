## Interaction States

### PricingActions.tsx — Pro Waitlist CTA

| State | Visual | Trigger |
|-------|--------|---------|
| Default (anonymous) | Button variant=secondary, "Join Pro Waitlist" label | Page load, not signed in |
| Default (signed in, not waitlisted) | Button variant=secondary, "Join Pro Waitlist" label | Page load, signed in, proWaitlist=false |
| Loading (API call in flight) | Button disabled + opacity-50 | Click while signed in, POST /user/waitlist pending |
| Confirmed (first join) | Inline text swap, no button: "You're on the list! We'll let you know when Pro launches." in --success color | POST /user/waitlist 200 response |
| Confirmed (return visit) | Same inline text as above, no button | Page load, proWaitlist=true from /user/plan |
| Error (API failure) | Small error text below button: "Something went wrong. Please try again." in --error color | POST /user/waitlist non-200 response |

**Transition:** Confirmation text fades in using existing `fade-in` keyframe (opacity 0 -> 1, translateY 8px -> 0, 300ms ease-out-quart). No toast, no redirect, no modal.

### Anonymous user — Post-signup auto-enroll

| State | Visual | Trigger |
|-------|--------|---------|
| AuthModal opens | Existing modal animation (modal-enter keyframe) | Anonymous click on Pro CTA |
| After sign-in/signup | Page navigates to /pricing?waitlist=1 | callbackUrl contains ?waitlist=1 |
| Auto-enroll fires | No visible loading flash — fires immediately | PricingActions detects waitlist=1 param + session |
| Confirmation appears | Same inline text as authenticated join | POST /user/waitlist resolves |
| URL clean-up | ?waitlist=1 stripped from URL (replaceState) | After auto-enroll completes |

### Admin Users table — Waitlisted badge

| State | Visual |
|-------|--------|
| User is waitlisted | Badge with lime-green background (#84cc16) + dark text (#3f6212), label: "Waitlisted" |
| User is not waitlisted | No badge change — plan and role badges unchanged |

### Admin Analytics — Waitlist stat card

| State | Visual |
|-------|--------|
| Loading | Skeleton h-28 rounded-xl (matches existing stat card skeletons) |
| Data loaded | New StatCard with label "Pro Waitlist" and integer count |
| Zero waitlist | StatCard shows "0" — no empty state substitution |

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | "Join Pro Waitlist" |
| Waitlist confirmation (first join + return visits) | "You're on the list! We'll let you know when Pro launches." |
| AuthModal prompt context (anonymous Pro CTA) | No copy change to AuthModal — existing "Sign In" / "Create Account" headings apply |
| Error state (POST /user/waitlist failure) | "Something went wrong. Please try again." |
| Admin badge label | "Waitlisted" |
| Admin stat card label | "Pro Waitlist" |
| Admin users filter option | "Waitlisted" (added to plan filter dropdown as a filter value) |

Empty state: No dedicated empty state for waitlist. The admin /admin/users page already shows "No users found" via EmptyState component when filters return 0 results — the "Waitlisted" filter will surface this naturally.

Destructive actions: None in this phase. Joining the waitlist is irreversible but non-destructive (no confirmation required). No undo flow needed.

## Accessibility

- Confirmation paragraph uses role="status" (or equivalent live region) so screen readers announce the state change without a focus move.
- Loading state on Button sets aria-busy="true" while POST is in flight.
- "Waitlisted" Badge in admin table: wrap in `<span className="sr-only">Plan status: </span>` prefix if Badge lacks accessible label context.
- URL strip of ?waitlist=1 uses history.replaceState — no scroll or focus disruption.
- AuthModal already implements focus trap, scroll lock, and Escape key via Radix Dialog.
