<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> POST /user/waitlist | User-initiated API call from client; auth token validated server-side |
| Browser -> GET /user/plan | User-initiated API call from client; only returns owning user's data |
| URL query param ?waitlist=1 | Untrusted input from URL; triggers auto-enroll only when combined with valid authenticated session |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-07 | Spoofing | ?waitlist=1 auto-enroll | mitigate | Auto-enroll only fires when `isSignedIn` is true AND `tier.key === "pro-waitlist"` AND `searchParams.get("waitlist") === "1"`. The actual enrollment requires a valid JWT via authFetch -- the query param alone does nothing. |
| T-03-08 | Tampering | callbackUrl | accept | callbackUrl is a hardcoded string `/pricing?waitlist=1`, not user-supplied. AuthModal redirects to this fixed path after auth. |
| T-03-09 | Information Disclosure | proWaitlist in /user/plan | mitigate | proWaitlist is only read from the authenticated user's own /user/plan response. No cross-user data exposure. |
| T-03-10 | Denial of Service | Double-mount auto-enroll | accept | React Strict Mode double-mounts effects. The POST endpoint is idempotent (Plan 01, T-03-05). Second call is a no-op. |
</threat_model>
